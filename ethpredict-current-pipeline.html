2 - Two lanes

Benchmark lane versus research trial lane

The benchmark answers "how good are the forecasters?" The meta-label path answers "when should we actually trade a forecast?" Both run inside the same v2 experiment.

Forecast benchmark

One run per benchmark model

Each model writes predictions and forecast metrics. Because route_to_meta_backtest is enabled, benchmark forecasts can also be sent through the same candidate and alpha backtest surface for secondary trade metrics.

Modelszero_return, momentum, lstm, timesfm
Trackio groupforecast_benchmark
Run namebenchmark/<run_id>/<model_id>
Meta-label MVP

One run per grid trial plus summary

Each trial retrains base LSTM folds, labels candidate trades with triple barriers, trains a meta-labeler, then backtests validation and test candidates. Selection is driven by validation net PnL with a minimum validation trade floor.

Grid size12 trials from base model, barrier, and alpha policy overrides
Trackio groupmeta_label_mvp
Run namemeta_label/<run_id>/<trial_id> and summary

Mental model: the base model predicts returns and direction; the candidate layer asks whether a forecast has enough edge after costs; the meta-labeler learns whether that candidate should be allowed to trade; the alpha backtest converts allowed candidates into trade rows and PnL metrics.

Selection contract

How a winner is chosen

The report keeps both the raw metric best and the trade-qualified best so an abstaining trial cannot silently masquerade as a tradable result.

Field Meaning Current config
Selection metric Metric path used to rank trials. metrics.validation.net_pnl
Selection mode Sort direction for the metric. max
Trade floor Minimum validation trades required to count as trade-qualified. min_validation_trades: 1
Test split Reported after selection; it should not drive trial choice. test_fraction: 0.2

5 - Code map

Where to read first

These are the small set of files that explain almost all of the current behavior.

src/experiments/staged_trial.py:718

Entry and dispatch

Loads the YAML config, detects v2, and routes the active path into the meta-label MVP.

src/experiments/meta_labeling_mvp.py:1220

Main v2 orchestrator

Creates the run directory, writes environment/config manifests, builds stage 0, runs benchmarks, expands the grid, runs trials, selects winners, and writes reports.

src/experiments/meta_labeling_mvp.py:914

Single trial body

Runs fold training, final model training, candidate creation, meta labeling, alpha backtests, diagnostics, and per-trial artifact writes.

src/experiments/forecast_benchmark.py:500

Forecast benchmark

Runs baseline, LSTM, and TimesFM forecast models over the same split and logs one Trackio run per model.

src/data/features_all.py:152

Raw data loader

Finds all split Lighter CSVs, concatenates them, drops duplicate timestamps, sorts them, and returns the price frame by granularity.

src/utils/trackio_logging.py:75

Trackio wrapper

Keeps tracking optional and non-fatal, flattens numeric metrics, JSON-safes paths/configs, and enables local GPU metric logging.

6 - Runbook

The commands that match this pipeline

Use these as orientation anchors. The first runs the full active v2 path; the second runs only the forecast benchmark entrypoint; the third opens the local Trackio dashboard.

Full v2 pipeline
uv run python -m src.experiments.staged_trial \
  --config configs/config.yml \
  --device cuda:0
Benchmark only
uv run python -m src.experiments.forecast_benchmark \
  --config configs/config.yml \
  --device cuda:0

Trackio dashboard:

uv run python -c "import trackio; trackio.show(project='ethpredict', open_browser=False, host='127.0.0.1', server_port=7860)"

CLI alternative: uv run trackio show --project ethpredict. The Python launch tends to be easier when sandboxed CLI networking is awkward.
